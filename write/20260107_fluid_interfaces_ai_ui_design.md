# 流动界面：AI 如何变革软件 UI 设计

用户界面正在经历一场深刻变革——从静态的"一刀切"设计，转向能够自适应用户的"流动界面"。设计师 Rio Lou 将 AI 驱动的 UI 未来描述为"像水一样流动"，去适配用户的真实思维方式。这种转变意味着：软件不再是僵硬的成品，而是一种响应式媒介——共享底层逻辑，但表面形态可以因人而异。

本文将探讨这一变革的几个核心维度：从静态 UI 到个性化 UI 的演进、AI 如何动态拆解和重组界面、设计师角色的转变（从像素工匠到系统架构师），以及 AI 作为通用界面范式的崛起。

## 从"一刀切"到个性化自适应 UI

传统用户界面基本上是"一刀切"的：设计师创建一套布局和工作流，期望适用于广泛的用户群体。用户需要学习界面结构，让自己去适应软件。这往往意味着：如果设计与用户的思维方式或特定需求不匹配，他们就会感到吃力。

AI 驱动的自适应 UI 颠覆了这种关系——不是用户适应软件，而是软件适应用户。正如 Rio Lou 所说：

> "有些人用要点思考，有些人用对话思考，还有些人需要把所有东西可视化地呈现出来。"

流动界面可以检测或被告知这些偏好，然后相应地改变形态。例如，AI 助手可能为分析型思维的用户呈现简洁的要点列表，而为视觉型思维者生成图表或时间线。

> "想象一下，AI 知道你更擅长用时间线处理复杂信息，或者你更喜欢快速的一来一往而不是长篇解释。"

同样的底层内容，可以作为交互式时间线、聊天对话、表格或画布来呈现——取决于什么格式"真正适合你的思维方式"，同时"基于相同的底层数据"运作。

### 自适应的实际效果

这种个性化不仅仅是简单的主题或布局设置；它是动态的、情境感知的。自适应界面可以利用 AI 实时学习用户行为和上下文。学术研究表明，根据用户专业水平调整复杂度的界面可以显著提升可用性。一项研究发现，渐进式展示高级功能的自适应 UI（对新手隐藏这些功能直到他们准备好）使新用户的任务完成速度提高了 35%，满意度评分也更高。

情境感知的适应（比如在用户使用移动设备且阳光强烈时简化布局，或在用户移动时放大按钮）同样能保持软件在各种条件下的可用性。简而言之，根据用户偏好和情境个性化界面可以减少认知负荷，让交互更高效、更自然。

Bret Victor 长期主张界面应该"匹配用户自己的思维过程，而不是强迫他们去适应僵化的界面"。Jakob Nielsen 则指出，AI 现在使界面能够完全不需要用户去迁就软件——AI 会解释用户的自然输入，甚至预测需求。他将旧范式（"用户必须适应系统的结构"）与新范式对比：

> "AI 驱动的界面可以解释自然语言、手势或上下文线索"来服务用户目标。

问题从"我怎么使用这个界面？"转变为"我想要达成什么？"，由界面智能地处理其余部分。界面开始变得"隐形"，因为它们实现了"恰到好处"的理想形态。

## AI 如何拆解和重组界面

实现这种适应性的关键是将传统界面分解为更细粒度的组件，AI 可以按需重新组合它们。在传统应用中，UI 布局和导航由设计师和开发人员硬编码。但现代 AI 系统（尤其是大语言模型和生成式模型）可以动态生成界面元素、对话框或可视化，以适应特定任务或用户意图。

与其说是固定的 UI，不如说是一个由用户意图和上下文引导的 UI 生成引擎。Google 最近关于 Generative UI 的研究捕捉到了这个思路：

> "这是迈向完全 AI 生成用户体验的第一步，用户自动获得根据其需求定制的动态界面，而不必从现有应用程序目录中选择。"

用户不再需要在不同的应用或工具之间做选择，只需描述需求，AI 就会即时组装界面和功能。

### 生成式 UI 的实例

在 Google Gemini AI 的实验中，用户询问穿搭建议时获得了一个非常视觉化、可浏览的"时尚搭配顾问"界面；而解释数学概念的查询则产生了一个交互式学习模块。AI 按需构建这些 UI，使用相同的核心引擎，但"理解向 5 岁孩子解释微生物组需要与向成人解释不同的内容和功能集"，或者"创建画廊需要与规划旅行完全不同的界面"。

传统的单一静态 UI 概念被动态 UI 层所取代，它可以根据需要重塑自己，从 UI 组件、可视化和交互模式的调色板中提取适合任务和用户的元素。

### 逻辑与呈现分离

这种 AI 主导的重组是通过将底层逻辑或数据与呈现层分离来实现的。现代软件越来越多地暴露其功能的 API 或语义表示，AI Agent 可以使用这些来组装新的 UI 视图。例如，电子邮件服务的核心逻辑（发送、排序、阅读邮件）可以通过经典收件箱 UI、语音助手或聊天机器人访问——都是同一后端上的不同"界面皮肤"。AI 可以选择哪种"皮肤"或 UI 元素组合适合用户当前的上下文。

Rio Lou 强调用"通用共享概念"来思考软件的力量，而不是固定的屏幕：

> "通过一组用于与 AI、Agent、代码、软件交互的通用共享概念……你可以将每一个变形以适应更多人和用例。每一个都可以利用最佳模型来做这件 UI 的事情，用最适合我的视图。"

换句话说，如果软件的能力是模块化且定义良好的（即"概念"），AI 就可以将这些能力重新打包成不同的界面形式——无论是为一个用户提供空白画布，还是为另一个用户提供分步向导——而无需更改底层代码。

### 现有工具中的实践

我们已经在一些现代工具中看到这种分解。许多生产力应用有多种模式或视图（比如项目管理工具让你在日历视图、看板视图或电子表格视图之间切换同一数据）。AI 更进一步，根据意图确定视图或即时组合一个新视图。

例如，Microsoft 的 Copilot 作为对话层跨 Office 应用进行中介——它可以将命令结果呈现为格式化的 Word 文档、PowerPoint 幻灯片或 Excel 数据表，取决于你的要求，有效地将输出重组为适合任务的界面。底层数据和逻辑保持共享，但呈现方式变得流动。

关键是，用户不再需要手动找到正确的界面或功能；AI 可以为他们做这件事。这与 Nielsen 的预测相呼应：

> "未来，AI Agent 将代表人类使用你的 UI……'用户'只与他或她的 Agent 交互，Agent 然后执行与所需服务的任何特定交互。"

在这种场景下，用户可以简单地发出请求（用自然语言），他们的 AI Agent 会找出要调用哪个应用或 UI 元素。在极端情况下，用户可能根本看不到传统 UI——只是通过聊天或合成视图获得的结果。

突破在于：

> "我们正在从'这是一个单一用途的内置界面，接受它吧'转向真正适合思维和团队运作方式的界面"，从简单和通用的东西成长为"捕捉个体思维美丽复杂性的形式"。

## 一个核心，多种面孔：共享逻辑与角色特定体验

流动界面的一个关键方面是：不同用户（或用户角色）可以拥有不同的界面体验，即使使用的是由同一底层系统驱动的同一软件。

过去，迎合不同角色往往意味着构建单独的产品或添加复杂的偏好设置。工程师可能使用文本密集的脚本界面，设计师使用可视化 GUI，项目经理使用仪表板——所有这些都是为重叠的领域服务。这种孤立的方式效率低下且割裂协作。

Rio Lou 的观点是：

> "如果你设计正确的概念而不是按钮，一套代码库可以服务所有人。"

换句话说，如果软件的基本能力是抽象设计的，表面 UI 可以为每个角色分化，而不需要为每个角色建立单独的代码库。系统检测（或配置）用户的角色、技能水平或意图，然后呈现定制的 UI 视图。

### Rio Lou 的亲身经历

Rio Lou 在职业生涯中亲身经历了碎片化工具的痛苦：

> "在过去 15 年左右，制作软件的艺术变得非常碎片化……每个角色使用自己的工具、自己的术语。"

设计师、工程师、产品经理最终在不同的软件孤岛中，相互交接产出物。在 Cursor（一个 AI 驱动的代码编辑器，Rio 是设计负责人），他正在扭转这种局面：

> "在 Cursor，事情有点翻转……设计对更多人来说是一个非常易于接近的概念，它把有设计抱负并想要构建东西的人聚集在一起。"

一个设计师和一个开发人员可以有效地使用同一个工具，但看到它的不同面向。例如，在 Cursor 中，非程序员最初被开发者中心的界面吓到（有"打开项目"、"连接到 SSH"等选项）。团队意识到他们可以简单地为这些用户提供一个 Agent 驱动的界面——本质上是一个空白的自然语言提示——并隐藏技术按钮：

> "作为初学者或非技术人员，我无法理解这些……所以我们就给你 Agent 视图空白，你可以直接开始做事情。"

底层能力（编辑代码、运行命令）仍然可用，但入口点被更改以适应用户。

### 灵活的界面包装

这种方法意味着表面 UI 变成围绕核心逻辑的灵活甚至临时"包装"，可以按角色或偏好更换。Rio 将其比作提供"相同事物的不同预配置和包装"，而不是创建完全分离的软件。

他用个人项目"RioOS"做了类比，在同一应用框架上重现了经典 MacOS、Windows 95 和其他操作系统主题：

> "如果你在它们之间切换……每一个都感觉非常真实，但实际上是同一个东西。"

这表明许多界面"皮肤"可以坐落在一组功能之上。信息是：UI 的许多差异是表面的惯例或样式；核心交互（打开文件、编辑内容、导航）是永恒的。流动设计利用这一点，允许多个 UI 呈现共存，每个都针对特定受众或上下文优化，但都连接到应用的相同底层模型。

### 消费软件中的早期迹象

我们在消费软件中也看到了共享逻辑/多 UI 的早期迹象。Notion 这个协作文档/数据库工具（Rio 是创始设计师之一）被设计为"工具箱"或"积木"应用——相同的组件可以配置成文档、表格、看板等。这种理念与流动界面一致：不同类型的用户（作家、项目经理、数据库管理员）都在 Notion 中工作，但配置各自的工作空间方式不同。

AI 可以通过自动化配置更进一步。例如，Notion 中的 AI 现在可以根据意图协助生成内容或构建页面（例如"创建一个带有季度里程碑的路线图"），有效地为你选择正确的模板或视图。

单一"通用"应用根据每个用例变形的概念正在出现。我们确实看到了所谓"万能应用"的崛起——工具足够通用且 AI 驱动，可以涵盖许多用途。ChatGPT 本身就是一个元示例：人们通过一个聊天界面将其用于写作、编码、研究、头脑风暴——尽管不是每种用途都有专门的 UI。但插件框架和扩展允许这个 Agent 在需要时以更结构化的方式呈现输出。

我们可以想象这样一个未来：在一个软件中，工程师可能切换到原始代码视图，而同一会话中的设计师看到相同逻辑的可视化布局——每个用户获得最能与他们对话的 UI，AI 在中间进行翻译。

### 传统 UI 与流动 UI 对比

| 方面 | 传统 UI（一刀切） | AI 驱动的"流动" UI（自适应/角色敏感） |
|------|------------------|--------------------------------------|
| 个性化 | 除了手动设置或主题外，所有用户的界面统一。用户必须遵从给定的设计和工作流。 | 界面根据个人用户偏好、认知风格和上下文动态调整。输出形式（文本、视觉、对话等）可以因用户而异。 |
| 界面组成 | 设计师预先确定的固定布局和导航流程。切换视图的模式有限（如果有的话）。 | UI 由 AI 在运行时使用模块化组件组成。系统可以根据用户的意图或任务生成或重新排列 UI 元素（例如从相同数据生成时间线 vs 图表）。 |
| 用户角色 | 通常为不同角色提供单独的工具或不同的模块（例如工程师 vs 设计师）——体验碎片化。 | 一个底层系统，为不同角色提供多个"面孔"。应用可以根据需要呈现程序员界面、自然语言助手或可视编辑器，都利用相同的核心逻辑。 |
| 设计师的任务 | 设计特定的屏幕和工作流，对单一预期用户旅程的 UI 元素进行像素级控制。 | 设计灵活的组件和样式规则系统。定义 AI 将用于生成和适应 UI 的模式和约束。更少关注静态模型，更多关注创建动态内容的容器和逻辑。 |
| 交互范式 | 用户必须学习和导航界面，用应用的术语发出明确命令（点击、菜单选择）。所有用户的一致性是关键。 | 用户可以用自然语言、语音、手势交互，系统解释意图。界面可能变得"隐形"或对话式，AI 主动呈现用户接下来需要的内容。一致性是个性化的而非统一的。 |

这个表格概括了共享底层平台如何在不失去共同基础优势的情况下实现不同的 UI 体验。正如 Rio 所说的激进赌注：如果在概念层面而不是形式层面设计，单一代码库或产品可以服务截然不同的受众。

## 设计师成为元设计师：从像素细节到模式与系统设计

在流动的、AI 组装的界面时代，设计师的角色正在演变。传统上，UI 设计师在像素级细节和固定布局上花费大量精力——决定精确位置、颜色、文案，并制作静态原型。随着 AI 接管更多界面组装工作，设计师正在转向定义 AI 将使用的高级设计语言、约束和模式。

从某种意义上说，设计师正在成为元设计师或系统设计师，策划界面的成分和规则，而不是手动组合每个屏幕。

### AI 集成的设计系统

一个明显的例子是 AI 集成设计系统的兴起。设计系统（可重用组件、样式指南和代码片段的集合）一直用于强制一致性。现在它们正在成为人类设计意图和 AI 生成之间的桥梁。

Figma 团队观察到：

> "随着 AI 承担更多设计工作，设计系统正在从为产品构建者编写转向也为 AI 编写。"

产品经理 Zoe Adelman 指出：

> "你可能期望设计师在没有明确写入系统的情况下推断的任何东西，对 AI 来说都是真正关键的上下文……AI 不了解品牌和业务上下文。"

这意味着设计师现在必须以 AI 可以理解的形式外化他们的知识和决策理由——例如，不仅编码调色板，还要说明何时使用每种颜色；不仅是一组组件，还有它们背后的意图。团队正在更新他们的设计文档，以捕获事物为什么以某种方式设计，并提供良好使用的示例，以便 AI Agent（比如"AI 设计助手"）可以遵循相同的原则。

Wayne Sun（Figma 设计师）将 AI 时代的设计系统描述为"创意身份的容器"：

> "它们彻底编码团队的品味，以便 AI 可以在开发的每个阶段应用它。"

换句话说，设计师的"品味"和愿景必须转化为 AI 的规则和训练数据，而不是直接转化为最终像素。如果做得好，AI 生成的界面不会看起来平淡或相同；相反，即使 AI 正在生成变体或新屏幕，每个产品的独特美学也将被保留。

这解决了一个主要担忧：没有强有力的护栏，AI 可能会产生不连贯或不吸引人的设计——Rio Lou 开玩笑地称之为"AI 垃圾"，如果你不给它一个观点的话。解药是设计师进行系统级控制：设置 AI 可以操作的约束，并将品牌的设计语言深深嵌入 AI 的知识中。

### 设计师工作流程的变化

实际上，设计师已经在调整他们的工作流程。有些人不再整天在 Figma 中制作静态模型，而是更多地在代码或基于文本的提示模板中工作。Rio Lou 提到：

> "我不整天坐在 Figma 里做模型。"

相反，他在写作（阐述设计想法）、用代码原型设计、素描之间交替，甚至"留一些空白空间让事情沉淀"以获得创意灵感。这种更流动的过程与设计整体系统或体验相一致，而不仅仅是屏幕。

设计师与开发人员的合作更加紧密，或者成为混合型"设计师-开发人员"，因为实现动态方面或训练 AI 可能需要脚本编写、数据集准备或组件编码。a16z 播客中 Rio 强调了设计和工程之间的界限正在模糊——设计师现在可以借助 AI 帮助"自己发布真正的软件"，工程师也关心 AI 输出中的设计选择。当 AI 可以从高级描述生成代码时，设计师的角色可能扩展到撰写这些描述（提示工程），然后策划/优化结果。

### 治理方面

设计团队正在扩大他们的职责范围，以治理 AI 在产品开发中的使用方式。设计系统团队现在不仅仅是生产组件贴纸表，还要考虑政策和护栏：例如，将道德准则或可访问性规则嵌入 AI 的约束中。

LinkedIn 的设计系统负责人 Grant Blakeman 指出，他们的范围已经扩展到：

> "治理产品如何构建……将设计规则和约束直接嵌入 AI 工具中，以便在创建时指导输出。"

这种主动指导意味着设计师实际上是在与 AI 共同设计——他们提供模式，AI 以新的组合实例化它们。因此，一些传统技能（如完善静态布局）在日常重要性上可能会降低，而其他技能（如架构自适应流程、制作多模态交互、理解 AI 行为）变得至关重要。设计师将需要了解 AI 模型如何做出决策和失败，以便他们可以围绕这些陷阱进行设计（例如，在 AI 不确定时提供后备 UI）。

### 抽象层级的提升

总的来说，设计师正在向更高的抽象层级移动。Apple 的人机界面组早在 2018 年的"设计流动界面"演讲中就触及了这个想法。他们强调设计界面要"感觉像是你思维的延伸"，通过与自然人类行为对齐来实现这一点。实现这一点意味着专注于原则（如响应性、与物理手势的连贯性），而不仅仅是漂亮的视觉效果。

> "不是我们的界面应该是流动的；而是我们是流动的，我们的界面需要对此做出响应。"
> —— Chan Karunamuni, Apple

这种理念现在延伸到 AI 驱动的个性化：设计师必须预见流动的用户需求范围，并确保系统可以流动地响应，由他们设定的设计模式引导。他们提供容器和乐高积木；AI（有时是用户自己）进行组装。这是从界面的唯一作者转变为动态界面的促进者或指挥者，界面可以以多种方式展开。

## AI 作为新的通用界面层

这种转变的另一个视角是将 AI 作为"通用界面"的概念，可以根据上下文以不同形式呈现。有了强大的 AI Agent，传统的逐应用界面模型开始解体。AI 本身成为用户和数字服务之间的中介，采取最适合当下的形式。

这可能是：
- 用户想要快速回答时的对话聊天
- 用户手忙时的语音界面
- 在文档中工作时的建议侧边栏
- 处理复杂任务时的完全重构 UI

底层原则是：用户向 AI 传达意图，AI 既确定要采取什么行动，也确定如何向用户呈现结果。

### Jakob Nielsen 的愿景

Jakob Nielsen 以一种挑衅的方式概括了这个愿景：

> "未来，AI Agent 将代表其人类使用你的 UI……人类用户只与他或她的 Agent 交互，Agent 然后执行与服务的任何特定交互以完成任务。"

在 Nielsen 的场景中，你可能告诉你的个人 AI，"规划我下个月去东京的旅行"，AI 在幕后与航空公司和酒店 UI 交互，比较选项，然后可能向你呈现一个整洁的摘要和选择——所有这些都不需要你手动点击每个网站。你体验的界面只是你与 AI 的对话（可能通过语音或聊天），加上它选择显示的任何摘要视觉效果（如行程）。

这是一个极端但越来越可信的情况：AI 成为主要界面，所有其他软件 UI 变成幕后的"服务"。在这样的世界里，一致性和熟悉感可能更多来自于你如何与你的个人 AI 互动（它的个性、它如何记住你的偏好），而不是任何单个应用的设计。

### AI 作为无处不在的界面层

即使没有完全的 Agent 范式，我们也看到 AI 功能正在成为跨应用的无处不在的界面层。考虑一下现在有多少现代应用嵌入了某种 AI 助手或聊天机器人：
- 文档编辑器（Notion 的 AI、MS Word 的 Copilot）
- 代码编辑器（GitHub Copilot、Cursor）
- 客户服务网站（到处都是聊天机器人）

这些通常使用类似的范式——提示框或聊天界面——本质上是通用输入方法。值得注意的是，尽管功能差异很大，许多软件产品正在趋同于聊天机器人或提示面板作为共同的界面元素。这表明自然语言正在成为一种用于命令软件的通用 UI。

然而，输出不必是文本。根据上下文，AI 可能输出一块文本、为你填写表单、生成图像或重组整个屏幕。例如，在设计工具中，AI 可能接受语音命令，然后重新排列你的画布或获取相关组件，以视觉方式显示结果而不是文本。

### 多模态融合

此外，AI 可以在界面中融合多种模态。具有语音和视觉模型的 AI 可能让你简单地看着某物并问，"这是什么？"——有效地将相机取景器加上语音作为你的界面，如 Google Lens 或 Apple 的 Live Text 结合 Siri 所见。AI 解释场景并以信息叠加方式响应。这些是 AI 如何使界面更加情境化和隐形的早期例子。

正如 LinkedIn/UXC 的趋势总结所指出的：

> "AI 将焦点从'我如何使用这个？'转移到'我想要达成什么？'……它超越了僵化的、基于规则的界面，创造流动的、自适应的、以人为本的体验。"

界面变得不再是一系列按钮，而更多是一种持续的对话或感觉自然的交互。在许多情况下，最好的界面可能根本不是传统 GUI——如果 AI 能正确直觉你的需求并直接做到（终极的"恰到好处"体验），用户甚至可能意识不到涉及复杂软件。

### 隐形界面与信任

Apple 的"恰到好处"设计理念和让技术透明化在这里一致：理想的 AI 驱动界面不会让用户思考界面本身。它是"隐形的"，用户专注于他们的目标，而不是机制。这已经在预测性功能中发生（例如基于电子邮件自动建议日历事件——在你选择接受或忽略建议之前不需要 UI）。

AI 在界面后台工作可以从用户旅程中删除步骤。然而，确保 AI 的行动保持可理解和可控很重要——完全隐藏的界面如果用户不知道 AI 做了什么或为什么，也可能令人困惑。因此，设计师面临新的挑战：设计 AI 作为界面的行为和反馈。这包括 AI 如何解释自己、用户如何给出更正，以及如何维持信任。

### AI 界面的几种形态

1. **对话 Agent（聊天/语音）**：用户界面就是对话（文本或口语）。例子：ChatGPT、Siri/Alexa、客户支持机器人。AI 解析用户请求并执行命令或获取答案，通常跨多个领域。这里的"UI"是最小的——一个文本框或语音提示——但可以通过连接后端处理许多任务。缺点是聊天对某些任务可能效率低下（导致长对话）。这就是为什么自适应系统可能在适当时从聊天切换到 GUI。

2. **应用内 AI 助手/Sidekick**：这些作为面板、工具提示或模态框出现在传统 UI 中，提供帮助。例如，Photoshop 的生成填充是侧边栏中的 AI 功能，接受文本提示并执行复杂的图像编辑，免去用户手动多步操作。在代码编辑器中，AI 可能内联悬停以建议代码。这种方法增强现有界面而不是替换它，但实际上，AI 正在观察上下文并进行干预。这是一个情境出现的新界面层（例如，变暗的建议或幽灵文本）。

3. **自动 UI 生成**：如前所述，像 Google 的生成式 UI 这样的系统可以为任务创建定制界面（比如用于规划活动的迷你应用）。这里 AI 扮演按需 UI 设计师+开发人员的角色。用户可能将此视为"神奇地"获得一个为他们特定任务量身定制的工具，仅使用一次或短期使用。这个概念可能导致短暂 UI——只存在于一个会话然后消失的界面——与今天的持久应用相当不同。

4. **Agent 式 UI（AI 代替用户操作）**：这是 Nielsen 的场景，AI Agent 导航现有界面。例如，Microsoft 的新 Copilot 可以像用户一样自主点击软件（他们演示了它从网站订购商品）。在这种情况下，AI 本身使用 GUI，并可能向你显示摘要。这是一个有趣的转折：AI 使用传统 UI，但人类看到的是它的更高级抽象。可访问性可以从中受益：AI Agent 可以为残障用户操作复杂的 UI，并呈现简化的反馈，有效地在不是为此设计的应用之上创建个性化的可访问界面。

### AI 作为统一层

总的来说，AI 在界面中的普及表明 AI 正在成为连接用户意图与功能的统一层，无论哪个应用或服务实际提供功能。这挑战了每个应用都有自己专用 UI 的旧观念。

相反，用户可能学习一个界面（与他们的 AI 助手交互），并通过它无缝访问数十个应用或服务。对用户体验来说，这可能意味着更少的上下文切换和认知负荷——如果你可以直接让你的 AI 处理，你就不必记住哪个应用做什么。这也意味着 AI 对你偏好的理解跨上下文携带（一种传统应用因保持单独的偏好设置而永远无法实现的个性化可移植性）。

## UX 的机遇与挑战

流动的、AI 驱动的界面的出现对用户体验设计有重大影响。积极的一面是，它承诺比以往任何时候都更加个性化、高效和包容的体验。挑战性的一面是，它引发了一致性、可预测性和设计师对体验控制的问题。

### 机遇与好处

**包容性和可访问性**：自适应界面可以实时调整以适应用户的能力和环境。例如，AI 可以自动简化 UI 并增加文本大小，如果它检测到老年用户或视力低下的人（而不是要求他们找到可访问性设置）。它可以为阅读障碍者提供语音交互，或为无法使用鼠标的人提供免提控制。这种个性化辅助方法可以使技术被更广泛的人群使用。可以想象 AI Agent 充当实时可访问性层，将任何应用的界面转换为适合用户需求（正如 Nielsen 建议的，"根据用户的特定需求转换内容和功能"）。这可以增强甚至替代一些手动可访问性适配，理想情况下改善访问。

**减少认知负荷和提高效率**：通过以用户偏好的格式呈现信息并预测需求，流动界面可以显著减少完成任务所需的步骤和心理努力。预测性 UI 元素（如建议下一个操作或自动填充信息）意味着用户花更少时间搜索菜单。研究表明，这种主动界面（例如推荐下一个要点击的按钮）通过简化工作流程"减少用户认知负荷并提高满意度"。对于专家用户，自适应 UI 可以在需要时呈现高级工具，加速高级使用，同时在不需要时隐藏它们，保持界面整洁。这种由 AI 管理的渐进式披露意味着初学者不会不知所措，专家也不会被拖累——这是 UX 中长期追求的理想，现在随着 AI 变得更加可实现。

**多模态和自然交互**：用户可以以最自然或最方便的模式交互——说话、打字、绘画或使用注视/AR——界面会适应。这可以使计算更加以人为本，利用我们的天生能力。Bret Victor 对"人类能力"的强调在这里找到共鸣。如果有人空间思维，也许 AR 界面（如 Dynamicland 的物理交互愿景）可以由 AI 召唤，让他们用手操纵数据。如果有人更喜欢故事，AI 可能会用叙事示例回应。这种流动性意味着技术可以在认知和身体上与人们相遇，而不是强迫每个人进入同一模式（例如指针+键盘或触摸屏）。最终，界面可能"感觉像是你自己的延伸……与你的思维同步"，正如 Apple 描述的目标。

**赋能创意用户**：用户获得更多权力来根据他们独特的工作流程定制软件，而不需要编程技能。过去，只有高级用户才能自定义软件（通过插件、宏等）。有了 AI，人们可以简单地告诉界面你想要什么。例如，"给我做一个显示 X 的仪表板"或"我更喜欢在时间线上看我的任务"，AI 会相应地重新排列。这民主化了一种设计形式——终端用户基本上可以通过高级命令设计自己的界面。这也可能刺激超专业化微工具的爆发（正如 Rio 的一位同事在 Twitter 上沉思的："AI 使软件创建变得如此容易，我们将看到超级利基、单一用途应用的爆发"，甚至是一次性使用的 UI）。虽然这听起来很混乱，但这也可能意味着对于每个特定问题，都有一个（即时生成的）优化解决它的界面。

**持续改进和学习**：AI 驱动的界面可以从每次交互中学习以变得更好。这意味着你使用系统越多，它就可以变得越简化，因为它学习你的习惯。例如，如果 AI 注意到你总是切换特定视图或忽略某个功能，它可以（在你的许可下）调整以淡化你不使用的东西，使你频繁的操作更快。这种自适应学习循环可以产生静态设计永远无法实现的真正个人化用户体验。它还为情境感知 UI 开辟了可能性——例如，你在工作中的仪表板在冲刺期和规划模式下自动变化，因为 AI 推断你在每种情况下需要什么。

### 挑战与考量

**失去一致性和可预测性**：经典可用性的基石是一致性——用户学习一次界面就可以预测东西在哪里。自适应 UI 本质上会变化，如果处理不当可能会让用户困惑。如果两个人比较笔记，他们的屏幕看起来不同（因为每个都是个性化的），这可能会使培训、支持甚至社交分享变得复杂（"我点哪里？我的看起来不一样"）。设计师需要确保仍有可识别的锚点或标准，也许让用户控制适应程度。一个可能的解决方案是有适应模式：例如，新手模式 vs 专家模式，这些是定义明确的状态，而不是可能更难跟踪的连续微妙变化。在流动性和稳定性之间取得平衡将是关键。

**透明度和信任**：当 AI 正在改变界面或代表用户执行任务时，用户需要信任它。如果 AI 决定隐藏一个功能或采取一个行动，用户怎么知道？缺乏透明度可能导致不信任或感觉失控。设计师必须纳入反馈机制——例如，AI Agent 可能解释，"我已经重新组织了你的工具栏，因为你最近没有使用这些工具；你可以撤销或在这里查看所有工具。"在不压倒用户的情况下提供一定程度的洞察是 UX 研究的活跃领域。还有错误的风险：AI 可能误解用户意图或上下文并呈现不恰当的界面，导致错误。健壮的测试和退出选项（例如，需要时恢复到熟悉的手动界面的方式）可以缓解这一点。

**设计和开发复杂性**：构建自适应、AI 驱动的界面比构建静态界面复杂得多。它需要投资于 AI 训练、上下文感知，并可能维护多种 UI 可能性。这可能增加开发成本和复杂性。此外，确保所有这些变体的质量并非易事——设计师必须想象许多状态和转换，本质上是设计模式而不是最终屏幕。存在"边缘情况"的风险，AI 呈现一些没有明确设计的奇怪或不可用的东西。需要严格的用户测试和持续改进，这改变了开发过程（变得更加数据驱动和持续，而不是一次性设计交付物）。

**设计师的角色和可能的替代**：当 AI 可以生成 UI 时，设计师（和前端开发人员）可能会担心他们的角色。AI 会简单地自动化界面创建吗？正在形成的共识是设计师不会被淘汰，但他们的重点会转移（如前所述）。他们将策划风格，确保可用性原则被编码，并处理高级架构。然而，有一个学习曲线——不是所有当前设计师都准备好编写提示脚本或训练 AI 模型。行业将需要提升技能或重新定义一些角色。积极的框架是设计师可以将重复工作交给 AI（如制作数十个变体或编码常规布局），专注于更高级的问题解决和创意策略。挑战是确保以人为本思维的独特价值不会丢失。设计师必须防范 AI 无意中产生偏见或排斥性的设计；人类监督对于维持道德和包容的设计标准至关重要。

**用户能动性和偏好**：虽然适应是有益的，但如果软件自己变化太多，一些用户可能会感到失去能动性。有一条微妙的界限在有帮助的适应和软件以可能令人恼火的方式"为你决定"之间。例如，想想 MS Word 的老 Clippy 助手——它试图通过弹出建议来帮助，但许多人发现它分散注意力。今天的 AI 能力强得多，但类似的陷阱存在。用户可能想要能够说，"不，我总是想看到 X 功能，停止隐藏它"，或"请停止切换到聊天模式，我更喜欢表单。"将用户偏好和反馈循环设计到自适应系统中是必不可少的，以便个性化保持以用户为中心，而不仅仅是算法驱动的。

**隐私问题**：为了有效适应，界面的 AI 需要收集用户行为、上下文、可能甚至个人信息的数据。这引发了隐私问题。用户可能对观察和亲密学习他们习惯的软件感到不舒服。需要明确的许可框架和设备端处理（在可能的情况下）来建立信任。还有数据偏见的担忧——如果 AI 基于许多用户的数据做出界面决策，它可能不会考虑独特的个人（经典的"我的界面开始假设我想要基于其他用户的东西，这可能不适用于我"问题）。理想情况下，个性化应该更多关注个人自己的数据，而不是某个全局平均值，否则它就失去了目的。

## 小结

尽管存在这些挑战，流动的、AI 驱动的界面的轨迹似乎很可能会加速。用户体验的好处——使技术更个人化、更直观、更有力——太大了，无法忽视。我们已经目睹了设计和人机交互研究中这个方向的范式转变。

Federico Francioni 的 UX Collective 对 SXSW 2025 的总结指出"AI 是新的 UI"，正是因为它"从根本上改变了我们与技术的交互方式……创造流动的、自适应的、以人为本的体验"。设计师和开发人员正在积极试验这些概念，用户开始期待更智能、更包容的界面，这要归功于他们在 Siri、Alexa、ChatGPT 和自适应网站等产品中的 AI 体验。

流动界面的概念——像水一样适应用户思维形状的软件——正在从愿景变为现实。Rio Lou 和其他人分享的想法描绘了一幅令人信服的画面：界面"弯向用户，而不是相反"。这是一个未来，我们的工具有共享的底层智能或逻辑，但可能有无尽的表面体验变体，针对每个用例、每个人、每个时刻塑造。

实现这个未来需要重新思考我们如何设计（关注系统和意图）、如何衡量成功（不是平均情况，而是跨情况的适应性），以及如何确保用户信任和能动性保持中心地位。转变已经在进行中：AI 正在渗透到界面设计的每个角落，从建议代码到重新格式化内容，再到按需生成全新的迷你应用。

对用户来说，这可能意味着技术真正与他们相遇——终于实现了计算增强人类能力而不强迫人类表现得像机器的长期梦想。对设计师和产品制造商来说，这意味着一个令人兴奋但充满挑战的新前沿：设计的不是单一界面，而是可以负责任地变形的灵活元界面。那些拥抱这种转变的公司（如提到的这些——Figma 的 AI 设计工具、Notion 的全能自适应工作空间、Apple 对自然交互的强调，以及无数用 AI 重新构想 UX 的初创公司）很可能会创造出让用户感觉神奇的产品。

终极目标是一种如此直观和响应的体验，界面本身淡入背景——或者正如最好的界面一直追求的那样，它"感觉恰到好处"。未来几年将是关键时期，我们将共同探索如何使界面真正流动，平衡设计艺术与人工智能的力量。
