播客摘要：对话 Cat - Claude Code 产品负责人

🚀 Claude Code 的诞生与产品哲学
  • Claude Code 最初是工程师 Boris 为了更好地理解公司 API 和探索软件工程增强能力而开发的个人项目。
  • 产品负责人 Cat 最初作为 20% 时间的贡献者，发现该工具能极大提升自己添加 RL 环境的效率，并提供了大量产品反馈。
  • 该工具在内部迅速传播（病毒式传播），从工程团队扩展到研究、数据科学、产品管理等技术邻近角色，在公开发布前就已获得高度认可。
  • 核心产品原则：
    1. 简洁至上 (Simplicity)： 利用终端（Terminal）作为极简的交互界面，迫使团队对展示哪些功能做出“残酷”的选择，避免用户初次使用时感到不知所措。
    2. 无缝上手 (Intuitive Onboarding)： 新功能应该无需复杂的引导教程，通过名称和一句话描述就能让用户直观理解并开始使用。
    3. 高度可扩展 (Extensibility)： 开发者环境千差万别，产品必须提供工具（如 hooks, 自定义斜杠命令, sub-agents）让每个团队都能进行定制，以适应复杂需求。
⚡ 高效的团队协作与开发流程
  • 团队拥有非常强大的“产品工程师” (Product Engineers)，他们对功能拥有端到端的自主权。
  • 开发流程： 大部分优秀功能源于工程师的原型创意 -> 发布到内部 Dogfooding 环境（千人规模试用）-> 收集直接反馈（是否易懂、有Bug、或备受喜爱）-> 快速迭代或决定公开发布。
  • 许多公开发布的功能都在内部经历了2-3轮迭代。同时，也有很多内部实验的功能最终没有上线。
  • PM的角色： 在开发者工具团队中，PM的角色是微妙的。主要职责包括：
    1. 设定更宏观的产品方向和设计原则（如，决定开放多大程度的定制化）。
    2. 负责产品的定价和包装 (Pricing and Packaging)，确保优秀的工程成果能被世界广泛使用。
    3. 对于大型项目（如IDE集成），会进行正式的产品审查流程；但对于小型功能（如To-do List），更侧重于快速原型和迭代，找到正确的设计形态。
  • 团队很少使用 Google Docs。产品文档通常是代码仓库中的 Markdown 文件，甚至直接记录在 PR 描述中。代码库本身就是“单一事实来源” (Source of Truth)。
  • Claude Code 降低了非工程师角色（如设计师、PM）贡献代码的门槛，团队设计师 Megan 也开始提交PR。
💬 用户反馈与产品迭代
  • 内部反馈渠道极其活跃：一个有上千名员工自愿加入的内部聊天室，平均每10分钟就会产生一条新的、高质量的反馈。
  • 团队极度渴求负面反馈，而非客套话。他们想知道产品哪里不好用。
  • 与约10家企业客户保持紧密合作，这些客户是外部最高信噪比的反馈来源。对于高优问题，团队承诺在一到两周内修复。
  • 使用 Claude Code 自身来处理反馈：通过集成 Slack 和 GitHub，可以快速综合分析不同渠道的类似反馈，并自动化处理如文档更新、关闭重复issue等任务。
📊 关于AI模型评测 (Evals) 的思考
  • 模型评测非常困难，团队关注两类评测，且两者都不完美。
  • 端到端评测 (End-to-end Evals)： 类似于在 SWE-bench 上运行测试，以确保对代码的修改不会降低整体性能。但缺点是粒度太粗，分数变化的原因不明显，需要深入分析日志。
  • 触发式评测 (Triggering Evals)： 评估模型在“何时”使用特定工具（如网络搜索）的决策能力。例如，不希望模型对每个问题都进行网络搜索。这类评测相对容易构建和评分，因为触发条件和结果是明确的。
  • 最难的是能力评测 (Capability Evals)，例如评估模型在“数据科学工作”上的表现是否提升，这需要复杂的测试环境和明确的“黄金标准答案”，很难实现。
  • 除了自动化评测，高度参与的用户社区是保持产品质量的最终防线，用户的直接反馈非常宝贵。
💡 给 PM 和用户的建议
  • 给 AI PM 的核心建议：最难得的技能是精准直觉——深刻理解当前模型的能力边界。要能判断一个功能构想是模型已具备80%能力（可通过提示工程补足），还是仅有10%（应等待3-6个月后再看）。
  • Claude Code 使用技巧：
    1. Demos not Docs (演示胜于文档)： 与其花时间写长篇文档，不如用 Claude Code 快速构建一个功能原型，亲身感受其价值。
    2. 把它当作一个热切的新手工程师： 当它犯错时，不要放弃，而是像指导新人一样直接给出反馈和纠正。它对反馈的接受度很高。
    3. 投资你的 `Claude.md` 文件： 这个文件是 Claude Code 的“记忆”，每次启动都会加载。把项目的架构、注意事项、测试方法等关键信息写进去，如同给新同事做 onboarding。
🔭 未来展望
  • 未来几个月的计划分为三大支柱：
  • 1. 核心CLI代理： 持续打造最强大的编码代理，并围绕其高度可定制性构建一个生态系统，让用户可以分享和一键安装各种配置。
  • 2. 发展SDK生态： 将 Claude Code SDK 推广为构建各类AI代理 (不仅仅是编码) 的最佳方式，支持法律、行政、健康、金融等领域的代理开发。
  • 3. 走出终端： 探索新的产品形态，让 Claude Code 的核心能力能够服务于更广泛的用户，包括数据科学家、产品经理、设计师，甚至市场和销售人员，降低使用门槛。
  • 由于模型技术迭代太快，做超过6个月的长期规划非常困难。团队更专注于为下一代模型构建产品。
